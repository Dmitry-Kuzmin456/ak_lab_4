## Язык программирования

Язык программы - ассемблер. Программа состоит из директив, меток, инструкций и
данных. Транслятор выполняет простую двухпроходную сборку: сначала вычисляет
адреса меток и размещение секций, затем генерирует бинарный код и debug dump.

### Синтаксис

Комментарии начинаются с `;` или `#` и продолжаются до конца строки. Пустые
строки игнорируются.

BNF-подобное описание:

```bnf
<program>      ::= { <line> }
<line>         ::= [ <label> ":" ] [ <statement> ] [ <comment> ]
<label>        ::= <identifier>
<statement>    ::= <directive> | <instruction> | <data-item>

<directive>    ::= ".section" ("text" | "data")
                 | ".org" <number>
                 | ".const" <identifier> <number>
                 | ".macro" <identifier> { <identifier> } <macro-body> ".endmacro"
                 | ".ifdef" <identifier>
                 | ".ifndef" <identifier>
                 | ".ifconst" <identifier>
                 | ".else"
                 | ".endif"

<instruction>  ::= <mnemonic>
                 | <mnemonic> <operand>

<operand>      ::= <number> | <char> | <identifier>
<data-item>    ::= ".word" (<number> | <char> | <identifier>)
                 | ".cstr" <string>

<number>       ::= decimal | hexadecimal-with-0x-prefix
<char>         ::= "'" <one-character> "'"
<string>       ::= "\"" { <character> } "\""
<identifier>   ::= letter { letter | digit | "_" }
```


### Семантика языка

Стратегия вычислений последовательная: инструкции выполняются в порядке роста
адресов памяти команд, пока поток управления не изменён командой перехода или не
выполнена остановка.

Все метки имеют глобальную область видимости. Метка в секции `text` обозначает
адрес байта в памяти команд, метка в секции `data` обозначает адрес слова в
памяти данных. Точка входа задаётся обязательной меткой `_start` в секции
`text`. Если `_start` отсутствует или объявлена не в `text`, транслятор
завершает сборку ошибкой.

Типизация минимальная и машинная:

- инструкция хранится в памяти команд как один или несколько байтов;
- слово данных хранится в памяти данных как 32-битное знаковое значение;
- символы представлены кодами Unicode/ASCII, которые затем выводятся младшим
  байтом;
- `.cstr` размещает каждый символ в отдельном 32-битном слове памяти данных и
  добавляет нулевой терминатор;
- непосредственные операнды для `*_IMM` занимают 1 байт;
- адресные операнды занимают 2 байта.

Макросы раскрываются текстово до разбора секций и меток. Поддерживаются
позиционные параметры:

```asm
.macro PRINT label
    OUT_CSTR label
.endmacro

PRINT msg
```

Условная компиляция работает по объявленным константам и макросам:

- `.ifdef NAME` --активен, если `NAME` определён как `.const` или `.macro`;
- `.ifndef NAME` - активен, если `NAME` не определён;
- `.ifconst NAME` - активен, если `NAME` определён;
- `.else` и `.endif` завершают условный блок.

## Организация памяти

Архитектура памяти - Harvard:

- память команд: `2^16` ячеек по 8 бит;
- память данных: `2^16` ячеек по 32 бита;
- память микрокода: 256 ячеек по 24 бита.

Память команд и память данных физически разделены в модели. Инструкции не
смешиваются со статическими данными и строками.

```text
Registers
+--------------------------------+
| AC IP AR DR CR BR PS           |
+--------------------------------+

Command memory, 8-bit cells
+--------------------------------+
| 0000 : opcode                   |
| 0001 : address high / immediate |
| 0002 : address low              |
| ...                             |
+--------------------------------+

Data memory, 32-bit cells
+--------------------------------+
| 0000 : .word / .cstr char       |
| 0001 : .word / .cstr char       |
| ...                             |
+--------------------------------+
```

Регистры процессора:

| Регистр | Размер | Назначение |
| --- | ---: | --- |
| `AC` | 32 бита | Аккумулятор, основной регистр вычислений и ввода-вывода. |
| `IP` | 16 бит | Instruction pointer, адрес следующего байта команды. |
| `AR` | 16 бит | Address register, адрес операнда в памяти данных или номер порта. |
| `DR` | 32 бита | Data register, буфер чтения/записи памяти данных. |
| `CR` | 8 бит | Command register, opcode текущей инструкции. |
| `BR` | 32 бита | Buffer register, второй операнд АЛУ. |
| `PS` | 4 бита | Флаги `N`, `Z`, `V`, `C`. |

Отображение сущностей языка на память:

- инструкции секции `.section text` размещаются в памяти команд;
- статические данные секции `.section data` размещаются в памяти данных;
- `.org` задаёт следующий адрес размещения в текущей секции;
- `.word` занимает одно 32-битное слово данных;
- `.cstr` занимает `len(string) + 1` слов данных;
- числовые литералы в `LD_IMM`, `ADD_IMM`, `SUB_IMM`, `CMP_IMM` кодируются в
  памяти команд одним байтом;

## Система команд

Архитектура команд - аккумуляторная CISC. Есть команды разной длины:

- 1 байт: безоперандные инструкции;
- 2 байта: инструкции с 8-битным immediate;
- 3 байта: инструкции с 16-битным адресом или номером порта.

Это важно для CISC-классификации: выборка инструкции явно читает разное число
байтов из памяти команд. Дополнительно реализована сложная инструкция
`OUT_CSTR addr`, которая одной ISA-командой выводит null-terminated строку из
памяти данных через порт `0`. Внутри она исполняется микрокодовым циклом:
прочитать слово, проверить `0`, вывести младший байт, увеличить `AR`, повторить.

### Кодирование

Байт opcode всегда первый. Для 16-битных адресов используется big-endian порядок:
`opcode`, `addr_high`, `addr_low`. Для immediate используется `opcode`,
`imm8`.

| Инструкция | Opcode | Длина | Семантика |
| --- | ---: | ---: | --- |
| `HLT` | `0x01` | 1 | Остановить модель. |
| `CLA` | `0x02` | 1 | `AC <- 0`. |
| `NOP` | `0x03` | 1 | Нет операции. |
| `LD addr` | `0x04` | 3 | `AC <- DataMem[addr]`. |
| `LD_IND addr` | `0x05` | 3 | `AC <- DataMem[DataMem[addr]]`. |
| `LD_IMM imm` | `0x06` | 2 | `AC <- imm8`. |
| `ST addr` | `0x07` | 3 | `DataMem[addr] <- AC`. |
| `ST_IND addr` | `0x08` | 3 | `DataMem[DataMem[addr]] <- AC`. |
| `ADD addr` | `0x0A` | 3 | `AC <- AC + DataMem[addr]`. |
| `ADD_IND addr` | `0x0B` | 3 | `AC <- AC + DataMem[DataMem[addr]]`. |
| `ADD_IMM imm` | `0x0C` | 2 | `AC <- AC + imm8`. |
| `SUB addr` | `0x0D` | 3 | `AC <- AC - DataMem[addr]`. |
| `SUB_IND addr` | `0x0E` | 3 | `AC <- AC - DataMem[DataMem[addr]]`. |
| `SUB_IMM imm` | `0x0F` | 2 | `AC <- AC - imm8`. |
| `INC` | `0x10` | 1 | `AC <- AC + 1`. |
| `DEC` | `0x11` | 1 | `AC <- AC - 1`. |
| `BEQ addr` | `0x12` | 3 | `IP <- addr`, если `Z = 1`. |
| `BNE addr` | `0x13` | 3 | `IP <- addr`, если `Z = 0`. |
| `BLT addr` | `0x14` | 3 | `IP <- addr`, если `N = 1`. |
| `BGT addr` | `0x15` | 3 | `IP <- addr`, если `N = 0` и `Z = 0`. |
| `JMP addr` | `0x16` | 3 | `IP <- addr`. |
| `CMP addr` | `0x17` | 3 | Выставить флаги для `AC - DataMem[addr]`. |
| `CMP_IND addr` | `0x18` | 3 | Выставить флаги для `AC - DataMem[DataMem[addr]]`. |
| `CMP_IMM imm` | `0x19` | 2 | Выставить флаги для `AC - imm8`. |
| `IN port` | `0x1A` | 3 | `AC <- input_port[port]`. |
| `OUT port` | `0x1B` | 3 | `output_port[port] <- AC & 0xFF`. |
| `OUT_CSTR addr` | `0x1C` | 3 | Вывести C-строку из `DataMem[addr]` через порт `0`. |
| `MUL addr` | `0x1D` | 3 | `AC <- AC * DataMem[addr]`. |
| `DIV addr` | `0x1E` | 3 | `AC <- AC / DataMem[addr]`, целочисленно. |
| `MOD addr` | `0x1F` | 3 | `AC <- AC % DataMem[addr]`. |

`JUMP addr` поддерживается как псевдоним `JMP addr`.

### Такты и микротакты

Модель исполняется с точностью до такта. Каждый tick выполняет одну
микрокоманду или одну суперскалярную пару, если включён режим `--superscalar` и
текущий `IP` указывает на поддерживаемую пару.

Обычный fetch:

1. `CR <- CommandMem[IP]`
2. `IP <- IP + 1`, переход к выборке адреса или исполнению

Для 16-битного адреса дополнительно:

1. `AR_H <- CommandMem[IP]`
2. `IP <- IP + 1`
3. `AR_L <- CommandMem[IP]`
4. `IP <- IP + 1`, переход к execute

Для immediate дополнительно:

1. `DR <- CommandMem[IP]`
2. `IP <- IP + 1`, переход к execute

Ориентировочные полные длительности с учётом fetch:

| Класс инструкции | Полный цикл |
| --- | ---: |
| `HLT`, `CLA`, `NOP`, `INC`, `DEC` | 3 такта |
| `LD_IMM imm` | 5 тактов |
| `ADD_IMM imm`, `SUB_IMM imm`, `CMP_IMM imm` | 6 тактов |
| `LD addr`, `IN port`, `OUT port`, `JMP addr`, условный переход | 7 тактов |
| `ST addr`, `ADD addr`, `SUB addr`, `CMP addr`, `MUL addr`, `DIV addr`, `MOD addr` | 8 тактов |
| Косвенная адресация `*_IND` | на 1 такт больше прямой адресации |
| `OUT_CSTR addr` | 6 тактов выборки + 2 такта на каждый символ + 2 такта на нулевой терминатор |

## Транслятор

Транслятор запускается как консольная программа:

```bash
uv run python -m src.translator program.asm program.bin --debug program.dbg
```

Аргументы:

- `program.asm` -- входной текст на ассемблере;
- `program.bin` -- выходной бинарный файл;
- `--debug program.dbg` -- необязательный человекочитаемый дамп машинного кода.

Этапы трансляции:

1. Очистка строк от комментариев и пустых строк.
2. Препроцессинг макросов и условной компиляции.
3. Разбор `.section`, `.org`, `.const`, меток, инструкций и данных.
4. Первый проход: вычисление адресов в `text` и `data`.
5. Второй проход: генерация памяти команд, памяти данных и бинарного файла.
6. Генерация debug dump, если указан `--debug`.

Формат debug dump:

```text
0000 - 040004 - LD max_i
0003 - 07000C - ST i
0024 - 04000C - LD i
0027 - 1D000D - MUL j
```

## Формат бинарного файла

Машинный код хранится в настоящем бинарном формате `AK4B`, а не в текстовом
виде.

```text
offset  size  description
0       4     magic bytes: "AK4B"
4       2     entry point, uint16 big-endian
6       2     command memory image length, uint16 big-endian
8       N     command memory image, bytes
8+N     2     count of initialized data words, uint16 big-endian
...     6*k   data items: uint16 address + int32 value, big-endian
```

Память команд сохраняется плотным префиксом до последнего использованного байта.
Память данных сохраняется разреженно: только явно инициализированные слова.

## Модель процессора

Модель процессора запускается как консольная программа:

```bash
uv run python -m src.machine program.bin --input input.txt --trace trace.log
```

Вывод в файл:

```bash
uv run python -m src.machine program.bin --input input.txt --output output.txt
```

Аргументы:

- `program.bin` - бинарник формата `AK4B`;
- `--input input.txt` - файл входного потока;
- `--output output.txt` - файл для результата моделирования;
- `--trace trace.log` - путь к журналу тактов;
- `--limit N` - максимальное число тактов, по умолчанию `1000000`;
- `--superscalar` - включить учебный суперскалярный режим.

## Запуск asm одной командой

Для запуска можно сразу собрать и исполнить исходный asm-файл:

```bash
uv run run_code program.asm --input input.txt --output output.txt --trace trace.log
```

Аргументы:

- `program.asm` - входной текст на ассемблере;
- `--input input.txt` - необязательный файл входного потока;
- `--output output.txt` - необязательный файл для результата моделирования;
- `--trace trace.log` - необязательный путь к журналу тактов;
- `--limit N` - максимальное число тактов, по умолчанию `1000000`;
- `--superscalar` - включить учебный суперскалярный режим.

Команда всегда сохраняет бинарный файл рядом с исходником: для `program.asm`
будет создан или перезаписан `program.bin`. Если `--trace` не указан, журнал
пишется рядом с исходником в `program.trace`. Если `--output` не указан,
результат программы печатается в терминал.

Схема процессора находится в [docs/proc_scheme.drawio](docs/proc_scheme.drawio).

### DataPath

DataPath содержит:

- память команд;
- память данных;
- регистры `AC`, `IP`, `AR`, `DR`, `CR`, `BR`, `PS`;
- АЛУ с операциями `ADD`, `SUB`, `MUL`, `DIV`, `MOD`, `INC`, `DEC`, `PASS`;
- входной и выходной stream buffers.

Основной поток данных:

1. ControlUnit выбирает источник шины (`Src`).
2. При необходимости выполняется чтение памяти команд или данных.
3. АЛУ вычисляет результат и обновляет флаги.
4. Результат записывается в выбранный приёмник (`Dst`).
5. При записи в память данных значение проходит через `DR`.

### ControlUnit и микрокод

ControlUnit микропрограммный. Микрокод хранится в отдельном массиве
`microcode_memory` на 256 ячеек. Каждая ячейка кодирует одну микрокоманду.

Формат микрокоманды:

| Поле | Биты | Назначение |
| --- | ---: | --- |
| `src` | 4 | Источник значения на внутренней шине. |
| `dst` | 4 | Приёмник результата. |
| `alu` | 4 | Операция АЛУ. |
| `mem` | 3 | Операция памяти. |
| `move` | 5 | Правило изменения `uPC`. |
| reserved | 4 | Резерв. |

Основные адреса микропрограмм:

| `uPC` | Назначение |
| ---: | --- |
| `0..1` | Fetch opcode и dispatch. |
| `2..5` | Выборка 16-битного адреса. |
| `6..7` | Выборка immediate. |
| `8..12` | Косвенная адресация. |
| `13` | `HLT`. |
| `14..15` | `CLA`, `NOP`. |
| `16..19` | `LD`, `ST`. |
| `20..31` | `ADD`, `SUB`, `CMP` и immediate-варианты. |
| `32..34` | `INC`, `DEC`, `JMP`. |
| `35..40` | Ветвления и `IN`/`OUT`, часть логики в ControlUnit. |
| `41..42` | Цикл `OUT_CSTR`. |
| `43..48` | `MUL`, `DIV`, `MOD`. |

Ветвления и port I/O используют микрокоманду `FETCH`, а конкретное решение
принимается в ControlUnit:

- для `BEQ/BNE/BLT/BGT` проверяются флаги `PS`;
- для `IN` проверяется номер порта и наличие данных во входном буфере;
- для `OUT` младший байт `AC` добавляется в выходной буфер;
- для `OUT_CSTR` используется специальный переход `CSTR_LOOP_OR_FETCH`, который
  либо завершает инструкцию на `0`, либо выводит символ и возвращается к чтению
  следующего слова строки.

### Журнал тактов

Trace пишется построчно и потоково, чтобы длинные программы не держали весь
журнал в памяти. Пример строки:

```text
tick=61 mode=superscalar issue=[0024: LD 000C, 0027: MUL 000D] parallel=1 pair=LD_MUL forwarding=LD_TO_MUL IP=002A AC=989010 PS=N0Z0V0C0
```

Для обычного микрокодового исполнения строка содержит:

- номер такта;
- режим `scalar` или `superscalar_fallback`;
- `uPC`;
- `IP`, `CR`, `AR`;
- `AC`, `DR`, `BR`;
- флаги `N`, `Z`, `V`, `C`;
- событие ввода-вывода, если оно произошло.

## Ввод-вывод

Вариант `stream` реализован через буферы:

- при старте модель читает весь файл `--input` в список кодов символов;
- `IN 0` извлекает один символ из входного буфера и кладёт его в `AC`;
- если входной буфер пуст, моделирование останавливается;
- `OUT 0` добавляет `AC & 0xFF` в выходной буфер;
- после завершения модель возвращает весь выходной буфер как строку.

Вариант `port` реализован специальными инструкциями `IN port` и `OUT port`.
Поддерживается порт `0`. Обращение к другому порту считается ошибкой модели.
`OUT_CSTR` также пишет в порт `0`, но номер порта в операнде не передаётся:
операндом является адрес C-строки.

## Строки

Вариант строк -- `cstr`, то есть null-terminated строки. Статические строки
задаются директивой `.cstr` в секции данных:

```asm
.section data
msg:
    .cstr "OK"
```

В памяти данных это размещается так:

```text
DataMem[msg + 0] = 'O'
DataMem[msg + 1] = 'K'
DataMem[msg + 2] = 0
```

Один символ занимает одно 32-битное слово памяти данных. Это неэффективно, но
так было удобнее реализовать в этой модели.

Строки можно выводить двумя способами:

- обычным циклом на asm через `LD`, `CMP`, `OUT`, `INC`;
- CISC-командой `OUT_CSTR addr`, которая выполняет такой цикл микрокодом.

## Суперскалярность

Усложнение `superscalar` включается флагом:

```bash
uv run python -m src.machine program.bin --superscalar
```

В учебной модели реализовано параллельное выполнение двух фиксированных пар
инструкций:

- `LD addr1` + `MUL addr2`;
- `LD addr1` + `CMP addr2`.

Обе пары встречаются в `prob1`, поэтому режим виден в реальной программе
варианта.

Механизм обнаружения:

1. Суперскалярная проверка выполняется только на границе ISA-инструкции, когда
   `uPC == 0`.
2. Модель декодирует первую инструкцию по текущему `IP`.
3. Если первая инструкция -- `LD`, декодируется следующая инструкция.
4. Если следующая инструкция -- поддерживаемая `MUL` или `CMP`, пара
   выполняется за один tick как одна issue-группа.
5. Если пара не распознана, выполнение продолжается обычным микрокодом.

Разрешение зависимости по данным:

- в паре `LD a` + `MUL b` значение `DataMem[a]` не записывается в `AC` отдельным
  тактом, а сразу используется как левый операнд умножения;
- в паре `LD a` + `CMP b` значение `DataMem[a]` сразу используется как левый
  операнд сравнения;
- в trace это помечается как `forwarding=LD_TO_MUL` или
  `forwarding=LD_TO_CMP`.

Пример trace:

```text
mode=superscalar issue=[0024: LD 000C, 0027: MUL 000D] parallel=1 pair=LD_MUL forwarding=LD_TO_MUL
mode=superscalar issue=[002D: LD 000E, 0030: CMP 000F] parallel=1 pair=LD_CMP forwarding=LD_TO_CMP
```

### Оценка прироста производительности

Для оценки был запущен алгоритм `prob1` в двух режимах: обычном микрокодовом и
с включённым `--superscalar`.

| Режим | Тактов | Результат |
| --- | ---: | --- |
| scalar | 707816 | `906609` |
| superscalar | 486392 | `906609` |


Снижение числа тактов:

```text
(707816 - 486392) / 707816 ~= 31.3%
```

В суперскалярном режиме было выполнено `15816` параллельных групп
`LD + MUL` и `LD + CMP`. Ускорение ограничено тем, что параллелятся только две
фиксированные пары инструкций, а остальные инструкции продолжают выполняться
обычным микрокодовым способом.

## Алгоритм варианта: `prob1`

Вариант `prob1` -- Project Euler problem 4: найти наибольший палиндром,
являющийся произведением двух трёхзначных чисел.

Программа находится в [prob1.asm](prob1.asm), golden-версия --
[tests/golden/prob1/source.asm](tests/golden/prob1/source.asm).

Алгоритм:

1. `i` идёт от `999` до `100`.
2. `j` идёт от `990` до `100` с шагом `11`.
3. Шаг `11` используется потому, что каждый шестизначный палиндром делится на
   `11`; значит, хотя бы один множитель должен быть кратен `11`.
4. Считается `product = i * j`.
5. Если `product <= best`, внутренний цикл можно завершить для текущего `i`.
6. Иначе число проверяется на палиндром:
   - `candidate` делится на `10`;
   - остаток через `MOD 10` добавляется к `reversed`;
   - `reversed = reversed * 10 + digit`;
   - цикл идёт до `candidate == 0`.
7. Если `reversed == product`, обновляется `best`.
8. После завершения перебора `best` печатается по цифрам через `DIV`/`MOD`.

Результат:

```text
906609
```

У алгоритма нет внешних параметров. Поэтому файл входного потока для этого теста
пустой, а результат всё равно передаётся через стандартный output stream
процессора.

## Тестирование

Golden tests запускаются командой:

```bash
uv run pytest
```

Каждый каталог `tests/golden/<case>/` содержит:

- `source.asm` - исходная программа;
- `input.txt` - входной поток;
- `expected_output.txt` - полный ожидаемый вывод;
- `expected_debug.txt` - обязательные строки debug dump;
- `expected_trace.txt` - обязательные строки trace;
- `limit.txt` - необязательный лимит тактов для длинных программ;
- `superscalar` - marker-файл запуска в режиме `--superscalar`.

Полный trace больших программ не хранится в эталоне. Вместо этого проверяются
репрезентативные строки: машинные инструкции в debug dump, ввод-вывод, отдельные
такты и события суперскалярного выполнения.

Покрытые сценарии:

| Тест | Что проверяет                                                                             |
| --- |-------------------------------------------------------------------------------------------|
| `hello` | `.cstr`, данные в data memory, `OUT_CSTR`, вывод строки.                                  |
| `cat` | Потоковый ввод через `IN 0`, вывод через `OUT 0`, остановка на пустом input.              |
| `hello_user_name` | Комбинация строк, ввода имени и вывода приветствия.                                       |
| `sort` | Условные переходы, сравнения и перестановку входных символов.                             |
| `double_precision` | 64-битную арифметику как пару 32-битных слов.                                             |
| `features` | `.org`, sections, labels, constants, macro, conditional, indirect addressing, `OUT_CSTR`. |
| `prob1` | Алгоритм Euler #4 и суперскалярные пары `LD + MUL`, `LD + CMP`.                           |
