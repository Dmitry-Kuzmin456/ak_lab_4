# Проверка схемы `new_scheme_4.drawio`

Схема стала читаемой: блоки показывают сущности процессора, сплошные линии
показывают движение данных, пунктирные линии показывают управляющие сигналы.
По ней можно объяснять исполнение команд на уровне микротактов, если рядом есть
ISA/микрокодовая спецификация.

Ниже я разбираю две команды из текущей ISA: `LD addr` и `ADD addr`.

## Команда `LD addr`

Семантика:

```text
AC <- DataMem[addr]
```

Команда занимает 3 байта в памяти команд:

```text
opcode, addr_high, addr_low
```

Потактово:

| Такт | Микродействие | Что видно на схеме |
| ---: | --- | --- |
| 1 | `CR <- CommandMem[IP]` | `IP` адресует `Command Memory`, сигнал `READ_CMD`, байт opcode попадает в `Instruction Registers` (`CR`). |
| 2 | `IP <- IP + 1` | `IP` идет через `SRC MUX`/`ALU`, результат возвращается через `Writeback bus` в `IP`. |
| 3 | `AR_H <- CommandMem[IP]` | снова чтение `Command Memory` по `IP`, старший байт адреса записывается в `AR`. |
| 4 | `IP <- IP + 1` | инкремент `IP` через `ALU` и `Writeback bus`. |
| 5 | `AR_L <- CommandMem[IP]` | чтение младшего байта адреса из `Command Memory` в `AR`. |
| 6 | `IP <- IP + 1`; dispatch execute | `IP` становится адресом следующей ISA-команды, `Control Unit` выбирает микрокод выполнения `LD`. |
| 7 | `AC <- DataMem[AR]` | `AR` адресует `Data Memory`, значение читается и через тракт записи попадает в `AC`. |

Итого: для `LD addr` схема объясняет весь путь: `IP -> Command Memory -> CR/AR`,
затем `AR -> Data Memory -> AC`.

## Команда `ADD addr`

Семантика:

```text
AC <- AC + DataMem[addr]
```

Команда тоже занимает 3 байта:

```text
opcode, addr_high, addr_low
```

Первые 6 тактов такие же, как у `LD addr`: выборка opcode, чтение двух байтов
адреса в `AR`, инкремент `IP` после каждого байта.

Потактово:

| Такт | Микродействие | Что видно на схеме |
| ---: | --- | --- |
| 1 | `CR <- CommandMem[IP]` | `IP -> Command Memory -> Instruction Registers`. |
| 2 | `IP <- IP + 1` | `IP -> SRC MUX -> ALU -> Writeback bus -> IP`. |
| 3 | `AR_H <- CommandMem[IP]` | старший байт адреса из `Command Memory` записывается в `AR`. |
| 4 | `IP <- IP + 1` | инкремент `IP`. |
| 5 | `AR_L <- CommandMem[IP]` | младший байт адреса из `Command Memory` записывается в `AR`. |
| 6 | `IP <- IP + 1`; dispatch execute | `Control Unit` по `CR` выбирает микрокод `ADD`. |
| 7 | `BR <- DataMem[AR]` | операнд читается из `Data Memory` и записывается в `BR`. |
| 8 | `AC <- AC + BR`, обновление `PS` | `AC` идет на левый вход через `SRC MUX`, `BR` на правый вход `ALU`, результат возвращается через `Writeback bus` в `AC`, флаги идут в `PS`. |

Итого: для `ADD addr` схема показывает ключевой аккумуляторный стиль:
левый операнд уже в `AC`, правый операнд загружается в `BR`, результат возвращается
в `AC`.

## Можно ли это однозначно восстановить только по схеме?

Не полностью. Схема корректно показывает состав процессора и основные маршруты
данных, но одна картинка не задает всю микросемантику. Из нее однозначно видно:

- есть Harvard-разделение `Command Memory` и `Data Memory`;
- `IP` адресует память команд;
- `CR / AR / DR` объединены в блок instruction/address registers;
- `AR` используется для доступа к памяти данных;
- `SRC MUX` выбирает левый вход `ALU`;
- `BR` является правым операндом `ALU`;
- результат `ALU` возвращается через `Writeback bus`;
- `PS` получает флаги, а `Control Unit` может использовать их для ветвлений;
- управляющие сигналы приходят из `MIR decoder`.

Но только по схеме нельзя однозначно восстановить:

- точную длину каждой ISA-команды;
- какие opcode имеют адрес, immediate или вообще не имеют операнда;
- порядок чтения `addr_high`/`addr_low`;
- точную микропрограмму и номера состояний `uPC`;
- какие операции поддерживает `ALU`;
- точные правила обновления флагов `N/Z/V/C`;
- специальные действия `IN`, `OUT`, `OUT_CSTR`, условных переходов;
- суперскалярные пары `LD+MUL` и `LD+CMP`;
- разрядности регистров и памяти.

То есть схема полная как структурная схема datapath/control path, но не полная
как самостоятельная спецификация ISA и микрокода.

## Корректность схемы

В текущем виде схема выглядит корректной для объяснения модели из README:

- основные блоки на месте;
- направление данных согласуется с `LD`, `ST`, `ADD`, `CMP`, `JMP`, `IN`, `OUT`;
- отдельный `BR` нужен для двухоперандных операций ALU;
- `Writeback bus` хорошо объясняет запись результата в `AC`, `IP`, `AR`, `DR`,
  `BR`, `PS`;
- удаление большинства подписей не сломало смысл, потому что имена регистров и
  блоков остались внутри сущностей.

Есть несколько мест, которые лучше держать в тексте рядом со схемой:

- `Instruction Registers` на схеме содержит `CR / AR / DR`, но детали `AR_H` и
  `AR_L` не показаны; это нормально для читаемости, но важно в описании fetch.
- `IN/OUT` в реализации частично обрабатываются `Control Unit`, а не только
  проводами datapath; схема показывает порты, но не раскрывает эту специальную
  логику.
- `OUT_CSTR` является микрокодовым циклом, и из схемы видно наличие пути
  `Data Memory/DR -> Ports OUT`, но не видно условия остановки по нулевому
  терминатору.
- Суперскалярность лучше описывать отдельно: на структурной схеме она не
  читается однозначно и может перегрузить картинку.

Вывод: схема сейчас хорошая и читаемая. Для отчета ее можно использовать как
основную структурную схему процессора, а потактовое поведение команд описывать
текстом рядом, как выше.
